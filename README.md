# projet-11

Vous êtes développeur front-end au sein de l’agence Webgencia. 
Votre client Qwenta souhaite faire développer un “Menu Maker” : un site permettant aux restaurateurs d’afficher et de mettre en page leurs menus facilement, en quelques clics. 
 
 
Soufiane, le Product Owner du projet est en congé maladie. Il vous a chargé, exceptionnellement, de préparer et planifier ce projet. Il prendra le relai pour présenter votre travail au client.
Il vous a envoyé la maquette du site et ses spécifications fonctionnelles.

Et il vous a préparé un brief :
Hello,
Merci de prendre ce projet en mon absence.
Tu vas voir, la gestion de projet est intéressante et importante aussi pour les devs, même s’il n’y a pas de code !
Voici les documents que tu devras réaliser :
1) Le document de spécifications techniques du Menu Maker afin d'expliquer toute la stratégie (hébergement, sécurité...)
Je te mets un modèle en pièce jointe. 15 pages max suffiront mais n’oublie pas d’expliquer tes choix.
2) Un tableau Kanban finalisé, réalisé sur un outil du type Notion
Je te mets en pièce jointe une carte des besoins et attentes des utilisateurs. Les cartes Notion dans ce tableau sont des User Stories (US) (le "quoi" du point de vue utilisateur).
Le Kanban que je veux que tu réalises, lui, doit contenir des tâches techniques (le "comment" du point de vue dev).
Il faut donc prendre chaque US et la découper.
Par exemple, l'US "En tant qu'utilisateur, je veux créer un compte" se transforme en plusieurs tâches dans ton Kanban :
- Front-end : Création du formulaire d'inscription (HTML/CSS/Validation).
- Back-end : Création de la route API POST /register et hashage du mot de passe.
- Back-end : Configuration de la base de données pour stocker les utilisateurs.
Ton Kanban sera donc bien plus fourni (environ 20 à 30 tâches) que ma liste initiale.
Chaque carte devra donc comporter :
plusieurs sous-tâches ;
une attribution à une personne (soit développeur front-end soit développeur back-end) ;
une estimation de la complexité en story points ;
une description claire des critères de succès (quand est-ce que la tâche est « terminée » ?) ; 
une rubrique « Spécifications techniques » (quels outils, quelle librairie, quel endpoint API…).
Tu devras aussi répertorier toutes les tâches du projet dans ton Kanban selon 4 catégories : à faire, en cours, à tester et terminé.
Merci et bonne journée, Soufiane.
Pièces jointes :
Cartes des besoins et attentes utilisateurs
Modèle spécifications techniques

Ce nouveau projet nécessite beaucoup de recherches pour s’assurer d'un travail de bonne qualité et aligné sur les dernières technologies !
Vous décidez donc, en parallèle de votre travail, de développer un système de veille.
La veille technologique est essentielle dans le métier. Elle permet de découvrir de nouvelles technologies pour enrichir votre travail.
Ces technologies sont ensuite utilisées pour répondre aux exigences techniques (livrable de document des spécifications techniques).
Ces spécifications techniques, une fois rédigées, servent à découper le projet en tâches spécifiques pour structurer le travail et suivre l'avancement du projet (livrable du tableau Kanban).
Sur l’outil Feedly ou Inoreader ou un autre outil de votre choix, vous organisez votre veille selon deux axes de curation :
Un premier axe qui se concentre sur les notions associées au projet du "Menu Maker". Par exemple SQL / No SQL ou encore React.Il vous aidera à réaliser votre document de spécifications techniques.
Un deuxième axe, plus général, sur le développement web : frameworks et bibliothèques populaires, accessibilité web et meilleures pratiques de sécurité en ligne.
Vous réunissez 5 à 10 sources d’informations pour chacun des ces 2 axes.
Elles doivent provenir de plusieurs supports (blogs, articles, réseaux sociaux, chaînes YouTube, podcasts…).
Quelques jours plus tard

Soufiane est de retour de congé. Vous avez une réunion avec lui pour lui présenter votre travail, avant qu’il ait une présentation avec le client.

Recommandations : 
Focalisez-vous dans un premier temps sur les user stories qui sont dans la priorité P1. Vous pouvez ensuite vous pencher sur les user stories des priorités inférieures (P2 et P3).
Créez systématiquement une description et une partie "Spécifications techniques" dans le résumé de votre ticket. Cela vous permettra de détailler les challenges techniques de chaque tâche.
Vous pouvez mettre en lien les tâches les unes avec les autres.

Points de vigilance : 
Même les développeurs expérimentés ont parfois du mal à estimer la difficulté/complexité d'une tâche. Ce qui est important ici est davantage la logique détaillée que le chiffrement final. 
Attention à être le plus précis possible. Par exemple, une tâche "Créer un menu" ne serait pas suffisamment précise, et ne serait donc pas une tâche réalisable.
